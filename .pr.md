## Assignment Summary
- Fixed open redirect handling in the authentication workflow by validating redirect targets before using `next` values in login and registration flows.

## Related Issue
- Closes #38

## Target Assignment Branch

## Design Note
- Added a shared helper that validates `next` values with Django's `url_has_allowed_host_and_scheme()`.
- Applied the validation consistently in both login and registration before redirecting users after authentication.
- Rejected unsafe external targets by falling back to the dashboard.
- Stopped reflecting unsafe `next` values back into hidden form fields or auth links, while still preserving safe internal navigation.

## Security Impact
- **Redirect Targets Validated**: Only same-origin redirect destinations are followed after login or registration.
- **External Destinations Rejected Safely**: Foreign hosts, protocol-relative URLs, and dangerous schemes like `javascript:` are ignored in favor of a safe default.
- **Legitimate Navigation Preserved**: Safe internal destinations such as `/charles/profile/` still work correctly across login and registration flows.

## Changes Made
- Added `get_safe_next_url()` in `charles/views.py` to centralize redirect validation.
- Updated `register()` and `user_login()` to use the validated redirect target instead of trusting raw `next` input.
- Updated `charles/templates/charles/login.html` and `charles/templates/charles/register.html` to preserve only safe internal `next` values across forms and auth links.
- Added redirect-focused tests for safe internal paths, rejected external targets, and fallback behavior.
